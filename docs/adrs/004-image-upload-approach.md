# ADR 004: 画像アップロード方式の選定

## ステータス

提案中

## コンテキスト

Astro移行（ADR 003）に伴い、従来のNext.js + Cloudinary管理画面による画像アップロード機能が利用できなくなった。
Astroは静的サイト生成（`output: 'static'`）を採用しており、サーバーサイド処理を含む管理画面をそのままAstroで再現することはできない。

ブログ記事に画像を掲載する際の新たなアップロード方式を検討する必要がある。

### 現状の課題

- 旧来のCloudinary管理画面（Next.js）はAstro移行後に削除された
- Cloudinary自体はまだ利用可能だが、アップロードの手段がない
- ブログ記事数は少なく、画像アップロードの頻度も高くない

## 検討した選択肢

### 選択肢1: リポジトリ直接配置

ブログ記事に使用する画像をリポジトリの `public/images/` 以下に直接コミットする。

**メリット:**
- 実装コストがゼロ
- 画像とソースコードのバージョン管理が一元化される
- 外部サービスへの依存がなくなる
- シンプルで運用が楽

**デメリット:**
- リポジトリサイズが増大する
- 画像の最適化（WebP変換、リサイズ等）は別途対応が必要
- Gitリポジトリに大容量バイナリを含めることへの懸念

### 選択肢2: CLIスクリプトでCloudinaryへアップロード

ローカルで実行するCLIスクリプトを作成し、Cloudinary APIを通じて画像をアップロードする。

**メリット:**
- Cloudinaryの画像最適化・変換機能を引き続き利用できる
- リポジトリに画像バイナリを含めない

**デメリット:**
- CLIスクリプトの実装コストが発生する
- Cloudinaryの認証情報（API Key/Secret）の管理が必要
- スクリプトのメンテナンスが必要

### 選択肢3: Astro hybrid モード + 管理ページ復活

Astroの`output: 'hybrid'`または`output: 'server'`モードに切り替え、サーバーサイド処理を有効にした上で管理ページを復活させる。

**メリット:**
- Webブラウザからアップロード操作ができる（旧来の体験に近い）

**デメリット:**
- Cloudflare Workersでのサーバーサイド処理が必要になり、Worker sizeの増大を招く（ADR 003で解決した問題が再発する可能性）
- 実装コストが高い
- 静的サイトのシンプルさが失われる

### 選択肢4: ローカル専用Node.jsサーバー

Astroとは完全に独立した、ローカル環境専用のNode.jsサーバーを用意する。このサーバーはCloudinaryへのアップロード機能を提供するが、Cloudflareにはデプロイしない。

**メリット:**
- Astroの静的サイト構成に一切影響しない
- Cloudinaryの画像最適化機能を利用できる
- ローカルのみで動作するため、認証情報の管理が比較的容易

**デメリット:**
- ローカル環境でのみ動作し、別ターミナルで起動する必要がある
- Node.jsサーバーの実装・メンテナンスコストが発生する
